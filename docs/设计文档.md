# Unitree G1 ROS2 架构重构设计

**日期**: 2026-07-04  
**状态**: 设计细化阶段  
**作者**: Claude + 用户协作设计

## 1. 概述

### 1.1 背景

当前项目目标是把基于 Unitree SDK2 / Rust CLI 的 G1 控制能力重构为 ROS2 架构，并继续保留 Pi Agent 语音交互能力。重构后的系统运行在 G1 板载计算机上，优先支持语音触发的安全行走、状态查询、TTS/反馈和可选 RealSense 视觉输入。

Unitree G1 的官方开发栈以 SDK2、CycloneDDS 和 ROS2 消息兼容为基础。官方 `unitree_ros2` 文档说明 SDK2 基于 CycloneDDS，ROS2 也使用 DDS，因此机器人底层可以直接用 ROS2 msg 通信，而不一定要把每个动作都包成 SDK API 调用。G1/H1-2 底层通信使用 `unitree_hg` IDL；ROS2 侧原生能力可以分为低层状态/控制 topic、`/api/*` 请求响应接口、Dex3 手部 topic 和语音/音频 API。部分 SDK2 示例或 DDS channel 可能使用 `rt/*` 命名，本文的 ROS2 默认配置以 `docs/unitree_ros2_topics.md` 中列出的 topic 为准，真机阶段再用 `ros2 topic list` 确认。

### 1.2 重构目标

- **主要目标**: 建立可维护的 ROS2 节点架构，明确 G1 原生 ROS2/DDS topic、`/api/*` 接口、项目内部 ROS2 接口和 Pi Agent 之间的边界。
- **核心原则**: 先保留现有语音控制能力，再逐步扩展视觉、手臂和低层控制。
- **安全优先**: 所有会让机器人运动的指令必须经过安全控制节点；高层运动、手臂控制和低层全身控制不能无门槛并发。
- **部署环境**: 默认全部部署在 G1 板载计算机；外部 PC 仅作为调试和可视化终端。
- **可验证性**: 每个阶段必须有可在真机或 `lo` 本地回环环境验证的验收命令。

### 1.3 关键决策

1. **采用 Unitree 官方栈**: 使用 `unitree_ros2` 的消息定义、`unitree_api` 请求响应消息和必要的 SDK2 通道能力。
2. **双接口模型**: 底层保留 Unitree 原生 ROS2/DDS topic 和 `/api/*` 请求响应接口，项目内部对上暴露稳定的 `/g1/*` ROS2 topic。
3. **高层运动优先**: P0 阶段优先使用 `/api/sport/request` 与 `/api/sport/response` 完成语音行走，不直接做全身低层控制。
4. **安全层独立**: 安全控制节点是唯一能把上层运动意图转成 G1 执行指令的入口。
5. **Pi Agent 保持独立**: 语音节点只做 ASR/视觉/ROS2 桥接，不把智能决策写进运动层。
6. **音频能力分层验证**: TTS、音量、播放和 LED 先按 `/api/voice/*` 抽象接入；ASR 文本输入仍按真机实际 topic 或外部 ASR 适配。

## 2. 官方约束与假设

### 2.1 已确认约束

- **ROS2 版本**: 官方 `unitree_ros2` 测试过 Ubuntu 20.04 Foxy，并以 Foxy 作为当前实机基线。
- **DDS 实现**: 使用 `rmw_cyclonedds_cpp`；Foxy 可能需要编译 CycloneDDS 0.10.x，Foxy 环境需按实际依赖构建。
- **网卡绑定**: `CYCLONEDDS_URI` 需要指定连接机器人的网卡，例如 `enp3s0`；离线仿真或本机测试可用 `lo`。
- **网络段**: 外部 PC 连接机器人时通常使用 `192.168.123.0/24` 手动地址；官方 ROS2 文档示例为 `192.168.123.99/24`。G1 具体板载 PC/MCU 地址以机器和固件为准。
- **G1 底层消息**: G1/H1-2 使用 `unitree_hg` IDL；低层状态/指令不是 Go2/B2/H1 的 `unitree_go`。
- **G1 原生 ROS2 topic/API**: `lowstate` 读取 500Hz 底层状态，`lf/lowstate` 读取低频状态，`/lowcmd` 做全身低层控制，`/arm_sdk` 做手臂 SDK 控制，`secondary_imu` 读取躯干 IMU；高层运动、手臂动作、语音/音频和模式切换通过 `/api/sport/*`、`/api/arm/*`、`/api/voice/*`、`/api/motion_switcher/*` 请求响应接口完成；Dex3 灵巧手使用 `/dex3/*/cmd` 和 `/lf/dex3/*/state`。
- **G1 高层运动 API ID**: `docs/G1_H1_API_Documentation.md` 已给出 sport 服务 ID：`7001` 获取 FSM ID，`7002` 获取 FSM mode，`7105` 设置速度 `JsonizeVelocityCommand{velocity: [vx, vy, vyaw], duration}`，`7110/7111` 切换用户/内部控制。

### 2.2 待真机确认项

- G1 板载计算机上 ROS2 topic 是否默认已发布，还是仍需显式 source `unitree_ros2/setup.sh` 并绑定具体网卡。
- `/api/sport/*` 在当前固件中的参数编码兼容性、FSM ID/FSM mode 取值和可用动作集合。
- `/api/voice/*` 在当前固件中的 TTS、音量、播放、LED 参数编码，以及 ASR 文本是否另有 topic。
- RealSense 是否接在板载 PC，USB 带宽和供电是否稳定。
- 机器人是否为 23 DoF、29 DoF 或带灵巧手版本，不同版本会影响电机索引和可控关节集合。

### 2.3 不做的假设

- 不假设所有 G1 固件都支持同一套高层运动动作。
- 不假设 `eth0` 一定是机器人 DDS 网卡。
- 不在 P0 直接允许语音触发 `/lowcmd` 全身低层控制。
- 不把 `unitree_ros2` 仓库中 Go2/B2 的 topic 名称直接套到 G1 上，除非真机验证通过。

## 3. 架构设计

### 3.1 整体分层

```text
┌─────────────────────────────────────────────────────────────┐
│ 应用层                                                       │
│                                                             │
│  ┌──────────────────┐        ┌──────────────────┐           │
│  │ 语音节点          │        │ 视觉节点          │           │
│  │ Pi Agent Bridge  │        │ RealSense Bridge │           │
│  └────────┬─────────┘        └────────┬─────────┘           │
│           │                           │                     │
└───────────┼───────────────────────────┼─────────────────────┘
            │ 运动意图/语音/图像          │ 图像/相机状态
┌───────────▼───────────────────────────▼─────────────────────┐
│ 中间层：安全控制层                                           │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ 安全控制节点                                           │  │
│  │ - 指令限幅、模式门控、频率限制、超时停止                 │  │
│  │ - 高层运动/手臂/低层控制互斥                             │  │
│  │ - 状态监控、异常拦截、审计日志                           │  │
│  └───────────────────────────────────────────────────────┘  │
└───────────┬─────────────────────────────────────────────────┘
            │ 安全后的控制指令
┌───────────▼─────────────────────────────────────────────────┐
│ 底层：G1 接口层                                              │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ G1 接口节点                                            │  │
│  │ - Unitree ROS2/DDS topic 订阅/发布                       │  │
│  │ - /api/* request/response 适配                           │  │
│  │ - 原生状态转项目内部 ROS2 topic                          │  │
│  │ - 语音/音频 API 与 ASR 输入适配                           │  │
│  └───────────────────────────────────────────────────────┘  │
└───────────┬─────────────────────────────────────────────────┘
            │ Unitree DDS / CycloneDDS
┌───────────▼─────────────────────────────────────────────────┐
│ G1 机器人运动控制单元 / 板载服务                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 接口边界

| 层级 | 面向谁 | 接口形式 | 稳定性要求 |
| --- | --- | --- | --- |
| Unitree 原生层 | G1 固件/SDK2 | `lowstate`、`/lowcmd`、`/api/*`、`/dex3/*`、可选 SDK2 DDS/RPC | 跟随官方和固件 |
| G1 接口节点 | 项目内部节点 | `/g1/state/*`、`/g1/robot/*`、`/g1/audio/*` | 项目内稳定 |
| 安全控制节点 | 应用层 | `/voice/cmd/*`、`/g1/safe_cmd/*` | 项目内稳定 |
| 应用层 | Pi Agent/相机 | Pi Agent API、ROS2 image/audio topic | 可替换 |

原则：Unitree 原生 topic 和 `/api/*` 请求只出现在 G1 接口节点中；应用层不直接发布 `/lowcmd`、`/arm_sdk`、`/dex3/*/cmd` 或 `/api/*/request`。

## 4. 节点详细设计

### 4.1 G1 接口节点

#### 职责

- 初始化 CycloneDDS/ROS2 环境，绑定正确网卡和 Domain ID。
- 订阅 Unitree 原生状态 topic，并发布项目内部标准状态 topic。
- 接收安全控制节点的安全指令，转换为 `/api/sport/*`、`/api/arm/*`、`/api/motion_switcher/*`、`/arm_sdk`、Dex3 或 `/lowcmd`。
- 适配 TTS、音量、音频播放、LED 和可选 ASR 等非运动能力。
- 提供硬件连通性、固件能力和 topic 可用性诊断。

#### Unitree 原生能力域

| 能力域 | 原生 ROS2 接口 | 消息类型 | 项目阶段 |
| --- | --- | --- | --- |
| 状态感知 | `lowstate`、`lf/lowstate`、`secondary_imu` | `unitree_hg/msg/LowState`、`unitree_hg/msg/IMUState` | P0 只读必选 |
| 低层全身控制 | `/lowcmd`、`lowcmd` | `unitree_hg/msg/LowCmd` | P0 禁用，P2 实验 |
| 高层运动 API | `/api/sport/request`、`/api/sport/response` | `unitree_api/msg/Request`、`unitree_api/msg/Response` | P0 主控制路径 |
| 模式切换 API | `/api/motion_switcher/request`、`/api/motion_switcher/response` | `unitree_api/msg/Request`、`unitree_api/msg/Response` | P1，人工确认 |
| 手臂动作 API | `/api/arm/request`、`/api/arm/response` | `unitree_api/msg/Request`、`unitree_api/msg/Response` | P1 |
| 手臂 SDK 控制 | `/arm_sdk` | `unitree_hg/msg/LowCmd` | P1，默认关闭 |
| Dex3 灵巧手 | `/dex3/left/cmd`、`/dex3/right/cmd`、`/lf/dex3/left/state`、`/lf/dex3/right/state` | `unitree_hg/msg/HandCmd`、`unitree_hg/msg/HandState` | P1/P2 |
| 语音/音频 API | `/api/voice/request`、`/api/voice/response` | `unitree_api/msg/Request`、`unitree_api/msg/Response` | P0 TTS/反馈，ASR 待确认 |

#### 能力域设计

- **状态感知域**: `lowstate` 是高频原始状态源，`lf/lowstate` 用于低频监控和 UI，`secondary_imu` 补充躯干 IMU。G1 接口节点必须把这些数据降采样、摘要化后再发布给项目内部节点。
- **高层运动 API 域**: `/api/sport/*` 是 P0 语音行走的唯一执行通道。接口节点负责维护 `sequence_id`、序列超时、返回码和 API ID 到内部动作枚举的映射。
- **模式切换域**: `/api/motion_switcher/*` 只服务调试、恢复和控制所有权切换。任何进入低层/debug/静音等影响运动安全的模式切换都必须有人机确认。
- **手臂与手部域**: `/api/arm/*` 负责手臂动作级能力，`/arm_sdk` 负责更直接的手臂关节控制，Dex3 topic 负责灵巧手。三者共享手臂/手部控制所有权，默认不与行走同时执行。
- **语音/音频域**: `/api/voice/*` 负责 TTS、音量、音频流、停止播放和 LED。ASR 文本来源不从该文档强推，P0 支持外部 ASR 或真机发现的 ASR topic 注入 `/g1/audio/asr`。

#### 项目内部发布 topic

- `/g1/state/health`: 接口节点健康、DDS 连通性、topic 可用性。
- `/g1/state/low`: 标准化低层状态摘要，来源为 `lowstate` 或 `lf/lowstate`。
- `/g1/state/imu`: IMU 姿态、角速度、线加速度。
- `/g1/state/motors`: 电机位置、速度、力矩、温度、lost 标志。
- `/g1/state/bms`: 电池电压、电流、电量、告警。
- `/g1/state/mode`: FSM、运动模式、控制所有权。
- `/g1/state/hand`: Dex3 手部状态摘要，P1/P2 启用。
- `/g1/audio/asr`: ASR 文本事件。
- `/g1/audio/status`: 音频服务状态。

#### 项目内部订阅 topic

- `/g1/safe_cmd/loco`: 安全后的高层移动指令。
- `/g1/safe_cmd/arm`: 安全后的手臂指令，P1 启用。
- `/g1/safe_cmd/hand`: 安全后的手部指令，P1/P2 启用。
- `/g1/safe_cmd/low`: 安全后的低层指令，默认禁用。
- `/g1/cmd/audio/tts`: TTS 文本。
- `/g1/cmd/audio/volume`: 音量控制。
- `/g1/cmd/audio/led`: LED/灯效控制。

#### 实现要点

- 启动参数必须包含 `network_interface`、`domain_id`、`robot_model`、`control_profile`。
- 启动时先执行能力探测：`lowstate`/`lf/lowstate` 是否有数据、`/api/sport/response` 是否可用、当前 FSM 是否允许运动、可选 Dex3/voice/arm topic 是否存在。
- 高层运动和低层控制使用互斥锁/控制所有权，禁止同一时间对机器人下发两类控制。
- 所有 `/api/*` 返回码、超时和当前 FSM 进入 `/g1/state/health`。
- G1 版本差异通过配置文件声明，不在代码里硬编码电机数量和索引。

### 4.2 安全控制节点

#### 职责

- 作为运动指令唯一入口，对所有语音/上层动作意图做限幅、模式检查和超时保护。
- 根据 `/g1/state/mode`、`/g1/state/imu`、`/g1/state/motors` 判断当前是否允许运动。
- 在异常时发布停止/保持指令，并拒绝新的运动指令。
- 为所有被修改或拦截的指令记录审计日志。

#### 订阅 topic

- `/voice/cmd/loco`: 语音节点产生的高层移动意图。
- `/voice/cmd/action`: 站立、停止、恢复等离散动作意图。
- `/g1/state/health`: 底层连接健康。
- `/g1/state/mode`: FSM 和控制所有权。
- `/g1/state/imu`: 倾倒、冲击、姿态异常检测。
- `/g1/state/motors`: 电机温度、lost 标志、异常速度/力矩。
- `/g1/state/bms`: 电池低压和供电异常。

#### 发布 topic

- `/g1/safe_cmd/loco`: 安全后的高层运动指令。
- `/g1/safe_cmd/stop`: 停止/保持命令。
- `/g1/state/safety`: 当前安全状态、拒绝原因、最近一次动作。
- `/g1/alert/emergency`: 紧急停止或人工介入提示。

#### P0 安全规则

- **速度限制**: `vx` 默认 `[-0.5, 0.5] m/s`，`vy` 默认 `[-0.3, 0.3] m/s`，`vyaw` 默认 `[-0.8, 0.8] rad/s`。
- **持续时间限制**: 单次运动命令必须 `0 < duration <= 5s`，超时后自动停止。
- **频率限制**: 同一来源运动指令默认不高于 10Hz，超频指令丢弃或合并。
- **模式门控**: 只有在明确允许运动的 FSM/高层控制模式下才转发移动指令。
- **连接门控**: `lowstate`/`lf/lowstate` 或 `/api/sport` 响应超过阈值无响应时进入安全暂停。
- **姿态门控**: IMU 姿态角、角速度、线加速度超过阈值时拒绝运动。
- **电机/电池门控**: 电机 lost、过温、电池低压时拒绝新运动。

#### 控制互斥规则

- `/api/sport` 高层移动控制与 `/lowcmd` 全身低层控制互斥。
- `/api/arm`、`/arm_sdk` 和 Dex3 手部控制共享手臂/手部控制所有权；与高层移动组合必须单独配置 `allow_arm_while_loco`，默认关闭。
- `/api/motion_switcher` 进入 debug/低层模式需要人工确认，不接受语音直接触发。
- 任一节点崩溃或安全状态未知时，默认进入停止/保持，不继续执行缓存命令。

### 4.3 语音节点

#### 职责

- 运行 Pi Agent 或连接 Pi Agent 运行时。
- 订阅 `/g1/audio/asr`，把文本、置信度、时间戳和可选图像上下文传给 Pi Agent。
- 将 Pi Agent 的动作决策转换成 ROS2 运动意图、TTS 和 UI/LED 反馈。
- 维护会话状态：唤醒、活动、等待、超时、取消。

#### 订阅 topic

- `/g1/audio/asr`: ASR 文本事件。
- `/camera/rgb/image_raw`: 可选视觉输入。
- `/g1/state/mode`: 当前机器人模式。
- `/g1/state/safety`: 安全状态。
- `/g1/state/health`: 系统健康。

#### 发布 topic

- `/voice/cmd/loco`: 高层移动意图。
- `/voice/cmd/action`: 停止、站立、恢复等离散动作意图。
- `/g1/cmd/audio/tts`: TTS 文本。
- `/g1/cmd/audio/volume`: 音量控制。
- `/g1/cmd/audio/led`: LED/灯效控制。

#### 会话规则

- 未唤醒时只处理唤醒词和停止词。
- 任何包含运动的意图必须显式带持续时间或由语音节点填默认短时长。
- “停止”“别动”“取消”类指令不需要完整会话，优先发布停止意图。
- Pi Agent 超时或失败时不重发运动命令，只可重发 TTS/反馈。

### 4.4 视觉节点

#### 职责

- 使用 RealSense SDK 或 ROS2 RealSense 驱动管理相机。
- 发布 RGB、深度、相机内参和相机健康状态。
- 提供断连重连、帧率降级和资源限制。

#### 发布 topic

- `/camera/rgb/image_raw`
- `/camera/rgb/image_compressed`
- `/camera/depth/image_rect_raw`
- `/camera/camera_info`
- `/camera/state`

#### 实现要点

- P0 不依赖视觉节点，视觉故障不能影响语音停止或运动安全。
- 默认 RGB `640x480@15fps`，确认资源充足后再升到 30fps。
- 图像只在 Pi Agent 需要时取最近帧，避免持续向大模型发送图片。

## 5. 数据流设计

### 5.1 语音行走流程

```text
用户语音
  ↓
G1 ASR 或外部 ASR
  ↓
/g1/audio/asr
  ↓
语音节点 → Pi Agent → 运动意图
  ↓
/voice/cmd/loco
  ↓
安全控制节点：限幅、模式检查、超时保护
  ↓
/g1/safe_cmd/loco
  ↓
G1 接口节点：/api/sport/request 高层运动 API
  ↓
G1 执行
```

### 5.2 紧急停止流程

```text
停止词 / 安全异常 / 节点超时
  ↓
安全控制节点进入 SAFE_STOPPING
  ↓
/g1/safe_cmd/stop
  ↓
G1 接口节点下发停止/保持
  ↓
/g1/state/safety 发布停止原因
```

### 5.3 视觉问答流程

```text
ASR 文本触发视觉需求
  ↓
语音节点读取最近 RGB 帧
  ↓
ASR 文本 + 图像上下文传给 Pi Agent
  ↓
Pi Agent 生成回答或动作意图
  ↓
回答走 TTS，动作仍必须经过安全控制节点
```

### 5.4 原生与内部 topic 映射

| Unitree 原生接口 | G1 接口节点输出/输入 | 用途 |
| --- | --- | --- |
| `lowstate`、`lf/lowstate` | `/g1/state/low`、`/g1/state/motors`、`/g1/state/imu`、`/g1/state/bms` | 状态读取和健康监控 |
| `secondary_imu` | `/g1/state/imu` | 躯干 IMU 补充 |
| `/api/sport/request`、`/api/sport/response` | `/g1/safe_cmd/loco`、`/g1/safe_cmd/stop` | P0 高层移动、站立、停止和交互动作 |
| `/api/motion_switcher/request`、`/api/motion_switcher/response` | 管理命令/人工确认流程 | 模式切换和控制所有权 |
| `/api/arm/request`、`/api/arm/response`、`/arm_sdk` | `/g1/safe_cmd/arm` | 手臂控制，P1 |
| `/dex3/left/cmd`、`/dex3/right/cmd`、`/lf/dex3/*/state` | `/g1/safe_cmd/hand`、`/g1/state/hand` | Dex3 灵巧手控制和反馈 |
| `/lowcmd`、`lowcmd` | `/g1/safe_cmd/low` | 低层全身控制，默认禁用 |
| 真机 ASR topic 或外部 ASR | `/g1/audio/asr` | 语音识别结果 |
| `/api/voice/request`、`/api/voice/response` | `/g1/cmd/audio/*`、`/g1/audio/status` | TTS、音量、播放、LED |

### 6.2 启动前检查

1. `ip addr` 或 `ifconfig` 确认 DDS 网卡名称。
2. `source /opt/ros/foxy/setup.bash`。
3. `source <unitree_ros2_install>/setup.bash`。
4. 设置 `RMW_IMPLEMENTATION=rmw_cyclonedds_cpp`。
5. 设置 `CYCLONEDDS_URI` 指向机器人网卡；本地仿真用 `lo`。

7. `ros2 topic echo /lowstate` 或 `ros2 topic echo lowstate` 确认状态刷新；不同 launch namespace 可能影响是否带前导 `/`。
8. `ros2 topic echo /api/sport/response` 并用只读/查询类请求确认高层 API 返回码链路。

### 7.2 网络策略

- 真机控制优先使用有线机器人控制网络，WiFi 只用于远程登录、日志和模型服务。
- CycloneDDS 只绑定机器人控制网卡，避免把 DDS multicast 发到公网或办公网。
- 本地仿真、单元测试和 CI 使用 `lo`。
- 多 ROS 环境共存时，不把 Unitree 的 setup 固化进全局 `.bashrc`；使用 launch/env 文件显式加载。

### 7.3 源码构建状态

Makefile 会固定 `unitree_sdk2` 和 `unitree_ros2` 的源码 revision，下载到
`.unitree/` 后分别执行 SDK2 CMake 安装和 ROS2 Foxy colcon overlay 构建。构建产物位于：

- SDK2：`.unitree/install/sdk2/`；
- Unitree ROS2：`.unitree/unitree_ros2/cyclonedds_ws/install-foxy/`；
- 项目 ROS2：`install-foxy/`。

对应入口为 `make unitree-source`、`make unitree-build` 和 `make foxy-build`。

## 10. 性能要求

### 10.1 延迟目标

- ASR 文本到安全指令发布: `< 200ms`，不含云端模型推理。
- 安全检查: `< 10ms`。
- 安全停止指令发布: `< 100ms`。
- 状态桥接: 低频摘要 `10-50Hz`，高频原始数据按需保留。
- 视觉帧获取: `< 100ms`，默认 15fps。

### 10.2 资源目标

- P0 不含视觉时总内存增量 `< 1GB`。
- 视觉启用后总内存目标 `< 2GB`。
- CPU 长期平均保持在板载计算机可接受范围，视觉和 Pi Agent 可降级。
- DDS/ROS2 视频 topic 不跨机器人控制网段外发。

## 11. 风险和缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| G1 固件接口差异 | topic/API 不一致 | 阶段 0 盘点，配置化 topic 和模式 |
| 高层/低层控制并发 | 机器人危险动作 | 安全节点控制所有权，默认禁用低层 |
| CycloneDDS 网卡配置错误 | 收不到状态或误发网络 | 启动前检查和 `/g1/state/health` |
| ASR 来源或 voice API 参数未确认 | 语音链路不完整 | P0 抽象接口，先支持外部 ASR 或 mock |
| 板载资源不足 | 延迟高、掉帧 | 视觉降帧、按需取图、服务拆分 |
| 源码构建依赖系统 ROS2 Foxy | 部署不可复现 | 明确 `/opt/ros/foxy` 前置条件并固定 Unitree revision |
